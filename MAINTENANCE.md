# 保守手順

このリポジトリを唯一の正本として扱います。公開サイトはmainブランチをGitHub Pagesで公開しています。通常の更新ではmainへ直接書き込まず、GitHubウェブ画面で専用ブランチとプルリクエストを使います。

## 用意するもの

GitHubにサインインできるスマートフォンまたはタブレットのブラウザだけで作業できます。ターミナル、パソコン、ローカルサーバー、Codespacesは必須ではありません。

パスワード、二段階認証コード、復旧コード、個人アクセストークン、APIキーを本文、コメント、コミット、AIへの依頼文へ書かないでください。

## 文章を更新する

1. GitHubでこのリポジトリを開きます。
2. mainから新しいブランチを作ります。名前は`update/2026-07-11-news`のように、目的と日付が分かる名前にします。
3. 対象ファイルを開き、編集ボタンから必要最小限を変更します。
4. 保存前に、変更先ブランチがmainではないことを確認します。
5. コミット後、Compare & pull requestからプルリクエストを作成します。baseがmain、compareが作成したブランチであることを確認します。
6. Files changedで、変更した行と対象ファイルだけが想定どおりであることを確認します。
7. Actionsのcheck-siteが成功するまで待ちます。失敗時はログの対象ファイルを確認し、同じブランチで修正します。
8. 公開してよいと確認した後だけ、Pull request画面からmainへマージします。
9. GitHub Pagesの公開完了後、公式サイトを開き、変更箇所、画像、リンク、スマートフォン幅の表示を確認します。

## 画像を追加する

画像の追加、再生成、描き直し、トリミング、色変更、縮小は、権利者が明示的に承認した場合だけ行います。

承認済みの元画像は保持します。追加前にファイル名、形式、表示する場所、alt、公開可否を確認します。追加後はプルリクエスト上で画像の差分とcheck-siteの結果を確認します。

## 取り消す

すでにマージした変更に問題がある場合は、元のプルリクエストを開き、Revertを選んで取り消し用プルリクエストを作成します。

取り消し用プルリクエストでもFiles changedとcheck-siteを確認し、内容が戻ることを確認してからマージします。GitHub Pagesが反映された後、公式サイトを再読み込みして復旧を確認します。

Revertが表示されない場合や競合がある場合は、新しいブランチで問題の変更だけを戻すプルリクエストを作ります。mainへ直接編集して急いで戻さないでください。

## 毎回確認すること

PUBLISH_CHECKLIST.mdを使います。設定、画像、LICENSE、ガイドライン、権利表記を変更する場合は、作業前に権利者の明示的な承認を確認します。

## 自動検査の位置づけ

check-siteは変更を報告するだけで、自動修正、自動コミット、自動マージ、公開判断はしません。検査workflowはGitHub Pagesの公開workflowとは別です。

GitHub Actionsが一時的に利用できない場合でも、公開済みサイトの表示は継続します。必須チェックやブランチ保護は、更新とRevertの手順を実際に確認してから検討します。
