# 鈴子ちゃん公式サイト 保守マニュアル

この文書は、鈴子ちゃん公式サイトの保守に必要な事実、依頼文、作業手順、確認項目、復旧方法を一つに統合した正本です。

AIへ保守を依頼するときは、このファイルと今回の依頼内容を最初に渡してください。別の保守説明書を新しく作らず、手順を変更するときはこのファイルを更新します。保守方法の変更履歴だけは`CHANGELOG.md`へ記録します。

## 1. 想定する環境

一般的な無料プランのチャットAIまたはコーディングAIと、エントリークラスのスマートフォンまたはタブレットだけでも保守できることを前提とします。

GitHubアプリまたはスマートフォン向けブラウザを使用します。パソコン、ローカル開発環境、有料API、有料サービスは必須ではありません。

AIがGitHubを直接読み書きできる場合は、調査、変更、検査、プルリクエスト作成、公開確認、報告まで一続きで実行します。直接読み書きできない場合は、AIが変更対象、変更後の全文または差分、確認手順を提示し、権利者がGitHubのウェブ編集画面で反映します。

小さい画面と限られた通信量を前提に、AIは画像ファイルを読み込まず、最初にこの文書を読み、次に依頼へ関係するテキストファイルだけを確認します。巨大な一括出力を避け、ファイル名、変更理由、確認結果を短く区切って扱います。

```text
今回の目的に必要な最小限だけを変更し、調査、変更、検査、プルリクエスト、公開確認、戻し方の報告まで進めてください。
今回の依頼内容:
ここへ依頼を書く
```

## 6. 最優先方針

現在正常に表示されているサイトを大きく作り直しません。

依頼された目的に必要な最小限のファイルと行だけを変更します。依頼と関係のない整形、言い換え、並べ替え、最適化、リファクタリングを混ぜません。

追加課金が必要なサービス、有料API、外部ホスティング、外部CMS、外部CDN、アクセス解析、広告、Cookie、外部フォーム、外部フォント、外部ライブラリを追加しません。

パスワード、個人アクセストークン、APIキー、二段階認証コード、復旧コード、秘密鍵、メール認証コードを要求、記録、表示、コメント、コミットしません。

AIは提案だけで止まらず、利用できる権限とツールの範囲でゴールまで進めます。ただし、検査に失敗した変更や、権利者の判断が必要な変更はmainへ反映しません。

## 7. 変更できる範囲

権利者から保守依頼を受けたAIは、次の作業を変更まで実行できます。

- 保守資料とチェック内容の整理
- 404ページなど補助ページの修正
- 存在しないローカル参照の修正
- 明らかなURL誤記の修正
- alt属性、noopener、noreferrerなどの不足を補う機械的修正
- Markdown、JSON、YAML、HTML、CSS、Pythonの単純な構文不良の修正
- 読み取り専用の自動検査の改善
- 読み取り専用のGitHub Actionsの改善
- 表示や意味を変えないアクセシビリティとセキュリティの修正

次は、権利者が対象と内容を明示した場合だけ変更します。不明な場合は変更せず、確認事項として報告します。

- 鈴子ちゃんの設定、プロフィール、本文の意味
- 利用ガイドライン
- `LICENSE`と権利表記
- 画像の追加、削除、差し替え、加工、変換、圧縮、リサイズ、切り抜き、ファイル名変更
- デザイン、色、余白、文字サイズ、配置、レスポンシブ表示、CSS
- `robots.txt`、`sitemap.xml`
- GitHub Pages設定、リポジトリ設定、ブランチ保護
- 外部公開先、公式URL、公式SNS

## 8. AIが最初に行う監査

1. 最新のmainと未処理のプルリクエストを確認します。
2. `MAINTENANCE.md`、`README.md`、`LICENSE`、`scripts/check_site.py`、`.github/workflows/check-site.yml`を確認します。
3. 今回の依頼に関係するファイルだけを追加で確認します。
4. 公開ファイルを変更する必要があるかを判定します。必要がなければ`index.html`、`404.html`、CSS、画像へ触れません。
5. 重複した説明、古い例外、存在しない参照、秘密情報、外部依存、表示変更の混入を確認します。
6. 変更対象と変更しない対象を確定します。

無料プランのAIで文脈量が不足する場合は、画像や全履歴を読み込まず、ファイル単位で確認します。推測で不足部分を補いません。

## 9. 標準の作業手順

1. 最新のmainを取得します。
2. 変更前のmainを指す復旧用ブランチを作成します。
3. 同じmainから一時作業ブランチを作成します。
4. 依頼された範囲だけを変更します。
5. `python3 -I scripts/check_site.py`を実行します。
6. 差分を確認し、依頼外の変更と公開表示の変更がないことを確認します。
7. 一つの目的だけを含むプルリクエストを作成します。
8. GitHub Actionsの`Site checks`が成功したことを確認します。
9. 原則としてSquash and mergeを使い、mainへ一つのコミットとして反映します。
10. GitHub Pagesと公式サイトを確認します。
11. 変更内容、検査結果、mainのコミット、復旧用ブランチ、公開結果、戻し方を報告します。

一時作業ブランチに複数コミットがあっても、mainでは一つのSquashコミットにまとめます。これにより、一回のRevertで今回の変更全体を戻せます。

AIがプルリクエストを作れず、権利者がmainへの直接反映を明示した場合だけ、復旧用ブランチを作成し、事前検査に成功した変更を一つのコミットとしてmainへ反映します。

## 11. スマートフォンまたはタブレットでの操作

GitHubアプリまたはブラウザでリポジトリを開きます。

AIがGitHubへ直接反映できない場合は、最新のmainから作業ブランチを作り、対象ファイルをウェブ編集します。AIにはファイル名と変更後の全文を出力させ、別のファイルへ誤って貼り付けないよう一ファイルずつ保存します。

プルリクエストを開き、次を確認します。

- Files changedが依頼したファイルだけになっている
- 画像、CSS、公開本文を変更しない依頼では、それらが差分に含まれていない
- ChecksまたはActionsの`Site checks`が成功している
- 追加課金、外部依存、認証情報が差分に含まれていない
- mainへの反映方法がSquash and mergeになっている

反映後に公式サイトを開き、トップページ、画像、リンク、スマートフォン幅、404ページを確認します。

https://abcderp2.github.io/suzukochan.officialsite/

404確認用URL

ローカル確認は任意です。端末性能や環境の都合で実行できない場合は、GitHub Actionsと公開サイトの確認を使用します。

## 12. 検査

標準検査

```bash
python3 -I scripts/check_site.py
```

`-I`はPythonを隔離モードで実行し、利用者環境の設定や追加パッケージの影響を減らします。

検査スクリプトは読み取り専用です。ファイルの自動修正、コミット、プッシュ、マージを行いません。エラーが一件でもあれば終了コード1を返し、mainへ反映しません。

ローカル表示確認を行う場合だけ、次を使用します。

```bash
python3 -m http.server 8000
```

ブラウザで`http://localhost:8000/`を開きます。終了するときは`Ctrl+C`を使用します。

## 13. 反映前チェックリスト

- 最新のmainを基準にしている
- 復旧用ブランチがある
- 依頼されたファイルだけを変更している
- 本文、画像、ガイドライン、LICENSE、権利表記を無断変更していない
- 表示変更なしの依頼では、公開HTML、CSS、画像の差分がない
- 有料サービス、外部API、外部依存を追加していない
- 認証情報が含まれていない
- `python3 -I scripts/check_site.py`が成功している
- 差分に競合記号が残っていない
- プルリクエスト本文に変更理由、表示への影響、検査、戻し方がある

## 14. 反映後チェックリスト

- GitHub Actionsが成功している
- トップページが表示される
- CSSと画像が表示される
- ページ内リンクと外部リンクが動作する
- スマートフォン幅で横スクロール、重なり、画像のはみ出しがない
- 存在しないURLで404ページが表示される
- main上のRevert対象コミットを記録した
- 復旧用ブランチを公開確認が終わるまで残している

## 15. 失敗時と元に戻す方法

マージ前に問題が見つかった場合は、プルリクエストを閉じます。mainは変更されません。

事前検査に失敗した場合はmainへ反映せず、同じ作業ブランチで修正して再検査します。

マージ後に検査または公開確認が失敗した場合は、対象のSquashコミットをRevertするコミットを作成し、再度検査と公開確認を行います。

復旧用ブランチは変更前のmainを指すだけのブランチで、公開には使用しません。復旧用ブランチでmainを強制上書きしません。

Revertが競合する場合は、復旧用ブランチと現在のmainを比較し、今回の変更だけを打ち消す新しいコミットを作成します。

force push、hard reset、履歴削除は使用しません。
